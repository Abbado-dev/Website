## Overview

ia-ide uses **CLI hooks** to get real-time visibility into what agents are doing. Instead of parsing terminal output, the CLI itself calls ia-ide's hook endpoint on every lifecycle event.

This is the foundation of ia-ide's dashboard features: status tracking, activity feed, notifications, and task progress all come from hooks.

## How It Works

When ia-ide spawns a CLI session, it:

1. **Creates a hook script** (`hook.sh`) that POSTs to `POST /api/runs/{run-id}/hook`
2. **Generates a hooks configuration** (`claude-hooks.json`) that maps all CLI events to the hook script
3. **Launches the CLI** with `--settings <path-to-hooks-config>`

The hook script is simple â€” it pipes stdin (the hook payload) to the ia-ide API:

```bash
#!/bin/bash
RESPONSE=$(curl -s -X POST "http://127.0.0.1:3000/api/runs/<run-id>/hook" \
  -H "Content-Type: application/json" -d @- 2>/dev/null)
echo "$RESPONSE"
```

## Hook Events

ia-ide registers handlers for all Claude Code hook events:

### UserPromptSubmit

**Trigger**: User sends a prompt to the agent.

**ia-ide behavior**: Marks the run as "running" if it was idle/queued. This is how ia-ide knows the agent is actively working.

### Stop

**Trigger**: Agent finishes responding to a prompt.

**ia-ide behavior**: Marks the run as "paused" (idle). If notifications are enabled for this run, sends a Discord notification with the last assistant message.

### StopFailure

**Trigger**: Agent encounters an error while processing.

**ia-ide behavior**: Marks the run as "failed". Sends a Discord notification with the error message.

### SessionEnd

**Trigger**: User quits the CLI or the session terminates.

**ia-ide behavior**: Marks the run as "completed". Sends a Discord notification if enabled.

### PreToolUse

**Trigger**: Agent is about to use a tool (Read, Edit, Bash, etc.).

**ia-ide behavior**: Persists an `agent.tool_call` event with the tool name and input. This powers the real-time activity feed in the dashboard.

### PostToolUse

**Trigger**: A tool call completed successfully.

**ia-ide behavior**: Persists an `agent.tool_result` event with the tool response.

### PostToolUseFailure

**Trigger**: A tool call failed.

**ia-ide behavior**: Persists an `agent.tool_result` event with the error message.

### Notification

**Trigger**: Claude Code sends a notification (permission prompt, idle prompt, etc.).

**ia-ide behavior**:
- Persists an `agent.needs_attention` event for the frontend to show a badge
- Sends a Discord notification for `permission_prompt` and `idle_prompt` types

### PermissionRequest

**Trigger**: Agent needs permission to perform an action.

**ia-ide behavior**: Does nothing â€” the user handles permissions directly in the terminal. ia-ide does not auto-approve.

### SubagentStart / SubagentStop

**Trigger**: Agent spawns or stops a subagent.

**ia-ide behavior**: Persists `agent.subagent_start` / `agent.subagent_stop` events for the activity feed.

## Hooks Configuration

The generated hooks configuration file maps each event to the hook script:

```json
{
  "hooks": {
    "UserPromptSubmit": [{ "hooks": [{ "type": "command", "command": "/path/to/hook.sh" }] }],
    "Stop": [{ "hooks": [{ "type": "command", "command": "/path/to/hook.sh" }] }],
    "StopFailure": [{ "hooks": [{ "type": "command", "command": "/path/to/hook.sh" }] }],
    "SessionEnd": [{ "hooks": [{ "type": "command", "command": "/path/to/hook.sh" }] }],
    "PreToolUse": [{ "matcher": "*", "hooks": [{ "type": "command", "command": "/path/to/hook.sh" }] }],
    "PostToolUse": [{ "matcher": "*", "hooks": [{ "type": "command", "command": "/path/to/hook.sh" }] }],
    "PostToolUseFailure": [{ "matcher": "*", "hooks": [{ "type": "command", "command": "/path/to/hook.sh" }] }],
    "Notification": [{ "hooks": [{ "type": "command", "command": "/path/to/hook.sh" }] }],
    "PermissionRequest": [{ "matcher": "*", "hooks": [{ "type": "command", "command": "/path/to/hook.sh" }] }],
    "SubagentStart": [{ "matcher": "*", "hooks": [{ "type": "command", "command": "/path/to/hook.sh" }] }],
    "SubagentStop": [{ "matcher": "*", "hooks": [{ "type": "command", "command": "/path/to/hook.sh" }] }]
  }
}
```

<Aside type="note">
Tool-related events (`PreToolUse`, `PostToolUse`, `PostToolUseFailure`, `PermissionRequest`, `SubagentStart`, `SubagentStop`) use `"matcher": "*"` to match all tool names.
</Aside>

## Codex Hooks

For Codex CLI, hooks are set up differently. Instead of `--settings`, Codex uses its own hook system installed via `ia_ide_adapters::codex::install_run_hooks`. The hook endpoint response format includes a `hookSpecificOutput` wrapper for Codex compatibility.

## Event Flow

```
CLI (claude/codex)
  â†’ hook fires (e.g., PreToolUse)
  â†’ hook.sh pipes JSON to POST /api/runs/{id}/hook
  â†’ ia-ide persists event to SQLite
  â†’ ia-ide broadcasts event via SSE
  â†’ Frontend receives event and updates UI
```

All hook events are persisted in the database and available via the [Events API](/docs/reference/api/#events).
